# 產品範圍與假設（給 BA / PO / 持份者）

> **目的**：用非實作細節嘅語言，寫清 **點解做、做畀邊個、點樣量度成功、做咩、唔做咩、預設前提** 同 **上線前最低驗收**，方便對齊期望、寫合約同排 UAT。  
> **實作與 API** 請配合 `USE_CASES.md`、`API_REFERENCE.md`、`ARCHITECTURE.md`。

---

## 1. 產品願景與定位

**Revision App** 係面向 Bootcamp／密集課程學員嘅 **Web 複習工具**：將講義與筆記上傳後，可用 **RAG 問答** 快速對照教材、用 **Quiz** 自測、用 **知識缺口** 睇弱項 topic、用 **Summary** 抓大綱，形成「問住學、做住練、睇住改」嘅閉環。

**與一般 LMS 嘅分別**：本產品 **唔** 取代完整學習管理（課程表、成績單、討論區）；重心係 **單人／小組溫書效率** 同 **AI 輔助理解**，前提是已有教材檔案（PDF／Markdown）。

## 2. 目標用戶與持份者（摘要）

| 類型 | 說明 |
|------|------|
| **主要用戶** | 需消化大量 PDF／筆記、希望有統一問答與自測入口嘅學員 |

| **部署／維運方** | 提供 MongoDB、OpenRouter、LlamaCloud 帳號與 Vercel（或同類）託管嘅負責人 |

角色關係圖與延伸討論見 [`STAKEHOLDER_MAP.md`](STAKEHOLDER_MAP.md)。

## 3. 痛點與解法（商業語言）

| 痛點 | 我哋點樣紓緩（產品層面） |
|------|--------------------------|
| 資訊量大、難消化 | 上傳後可 **聊天式** 針對段落發問，毋須逐份檔手動搜尋 |
| 教材散落多個平台 | 將 **本輪溫書用嘅檔** 集中索引，單一入口檢索（限已上傳內容） |
| 唔知邊度弱、缺乏自測 | **Quiz + topic 統計** 顯示錯得多的地方，方便安排下一輪溫習 |

詳細流程、例外路徑見 [`USE_CASES.md`](USE_CASES.md)。

## 4. 成功指標（KPI）

以下係 **建議目標**，實際達標需配合用戶調查或日誌抽樣；**技術門檻**（例如向量相似度過濾數值）屬工程調優，記於 [`ARCHITECTURE.md`](ARCHITECTURE.md)／[`NON_FUNCTIONAL_REQUIREMENTS.md`](NON_FUNCTIONAL_REQUIREMENTS.md)。

| KPI | 目標方向 | 建議量度方式 |
|-----|----------|--------------|
| 檢索／問答耗時 | 相對「純手動翻文件」減少顯著比例 | 小規模對照測試或用戶問卷 |
| 知識掌握感／Quiz 表現 | 同一文件重複練習後正確率或完成度提升 | `GET /api/quiz/stats` 趨勢（僅限本 App 內答題紀錄） |
| 格式覆蓋 | 常見課程格式（PDF、Markdown）可穩定 ingest | UAT 用真實講義樣本驗證 |
| 回答「跟教材」嘅可信賴度 | 用戶主觀評「有根據／離題少」 | 抽樣評分；工程上靠 RAG 過濾減少亂答 |

## 5. MVP 範圍（本版本交付內）

| 能力 | 用戶價值 | 對應文件（細節） |
|------|----------|------------------|
| 上傳 PDF／Markdown 並建立向量索引 | 教材可畀 RAG 使用 | UC-01、`USE_CASES.md` |
| 按教材內容聊天（串流） | 溫書問答 | UC-02 |
| 按文件自動出 MCQ、交卷、睇分同解析 | 自測與鞏固 | UC-03、UC-04 |
| Topic 層面知識缺口統計 | 知邊度弱 | UC-05 |
| 按文件生成大綱式摘要（串流） | 快速複習結構 | UC-06 |
| 睇已索引文件清單、刪單筆 | 管理索引、解決同名再上傳 | UC-07、UC-09 |

**刻意包含但非「教務系統」能力**：重置本機答題紀錄（UC-08）方便 demo／重測，**唔** 等同正式成績存檔或學籍管理。

## 6. 產品假設（預設成立，除非另行簽約）

1. **單一使用場景**：學員個人或小組溫書；**唔包含**多租戶隔離、機構級帳單分拆。  
2. **無登入／無角色權限**：任何可開啟部署網址嘅人皆可上傳、刪除文件（同一資料庫視角）。**唔適用**於需嚴格資料隔離嘅正式教務系統，除非外加認證層。  
3. **網路與配額**：OpenRouter、LlamaCloud、MongoDB Atlas 由部署方提供有效帳號；免費 tier 有配額上限，超出時行為以供應商為準。  
4. **語言**：介面與錯誤提示以 **繁中** 為主；Chat 會跟隨用戶輸入語言（見 `ARCHITECTURE`／system prompt）。  
5. **瀏覽器**：現代 Chromium／Safari／Firefox 最新兩個大版本；**唔保證** IE。  
6. **內容合規**：上傳者擁有或獲授權使用教材；本產品 **唔** 提供版權審查；敏感資料 **唔應** 上傳至無額外合規審查嘅公用部署。

## 8. 依賴與營運風險（持份者須知）

| 風險 | 對用戶嘅可能影響 | 緩解方向（產品／營運） |
|------|------------------|------------------------|
| 外部 AI／Embedding API 中斷或限流 | 聊天、Quiz、Summary 失敗或變慢 | 監控錯誤率；準備 API key 輪替；必要時升級供應商方案 |
| LlamaParse 解析失敗（掃描件過差、檔案過大） | 該 PDF 無法索引 | 提示改傳文字版或拆細檔；見 ingest 錯誤文案 |
| MongoDB Atlas 或向量索引未就緒 | 搜尋異常或無結果 | 部署 checklist：`MONGODB_VECTOR_SETUP.md` |
| 無登入導致 URL 洩漏即等同資料庫暴露 | 任何人可刪改上傳內容 | 僅限可信網路或加認證；PO 簽 scope 前須知 |

## 9. UAT／上線最低門檻（建議簽字前必跑）

> 詳細步驟見 `TEST_PLAN.md`；以下係 **最低 bar**，未通過唔建議對外宣稱「生產就緒」。

| # | 驗收項 | 判斷標準 |
|---|--------|----------|
| U1 | 上傳 | 至少 1 份 PDF 或 MD 成功 ingest，`chunkCount` > 0 |
| U2 | 聊天 | 同一文件相關問題有連貫回答（串流正常完結） |
| U3 | Quiz | 生成 → 全題作答 → 提交 → 結果頁顯示分數與解析 |
| U4 | 知識缺口 | 提交後 `GET /api/quiz/stats` 有合理聚合（或空列表行為符合預期） |
| U5 | Summary | 選文件後大綱可串流生成並可閱讀 |
| U6 | 刪除文件 | 刪除後清單更新；可再傳同名檔（409 解除） |
| U7 | 錯誤體驗 | 故意觸發 400／409／空文件時，用戶看到清晰提示（非空白頁） |

**技術閘門**（工程）：`npm run build` 通過、`npm run lint` 通過、`npm run test` 通過（見 `DEFINITION_OF_DONE.md`）。
