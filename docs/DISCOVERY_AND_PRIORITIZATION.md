# 需求發現與優先級（一頁敘事）

> **點解有呢個專案**：教材與筆記太多、太散，**難以快速搵到要用嘅內容**；若用 **AI 做統一檢索、問答同自測**，有望將 **搵資料＋首輪溫習** 嘅時間大約 **減半（~50%）**——呢個係 **待驗證目標**（見 `PRODUCT_SCOPE` KPI／UAT），唔係已發表研究結論。

---

## 1. 觀察到嘅問題（需求起點）

| 現象 | 對溫習嘅影響 |
|------|----------------|
| 檔案、平台多 | 花大量時間 **開錯檔、重複搜尋** |
| 單次課程資訊量大 | **唔知從邊段開始**，易拖延 |
| 缺乏即時自測 | **弱項遲先發現**，補救成本高 |

**一句話**：痛點唔係「冇教材」，而係 **「喺對嘅時間拎到對嘅一段」太難**。

---

## 2. 產品假設（點解用 AI）

- **統一入口**：本輪要溫嘅檔 **先上傳、再索引**，之後所有功能都建基於「已索引內容」。
- **問答（RAG）**：用自然語言 **定位段落意義**，減少逐頁翻。
- **Quiz + topic 統計**：用輸出倒逼 **主動回想**，並標出 **邊個 topic 弱**。
- **Summary**：用大綱 **縮短「先睇全貌」** 嘅時間。

**~50% 時間**：指 **主觀目標／假設**——實際要用戶測試或對照實驗量度；技術上透過 RAG 門檻等同工程品質，見 `ARCHITECTURE.md`。

---

## 3. 優先級（點解 MVP 係而家咁排）

| 優先 | 能力 | 理由 |
|------|------|------|
| **P0** | 上傳 → 索引 | 無內容即無後續 |
| **P0** | RAG 聊天 | 直接針對「快啲搵到答案」 |
| **P0** | Quiz 生成／提交／計分 | 閉環自測，易示範價值 |
| **P1** | 知識缺口（topic） | 建基於已有答題紀錄 |
| **P1** | Summary | 補「全貌」；可與 Chat 並行迭代 |
| **P0／P1** | 文件清單／刪除 | 營運同名檔、重傳；實務必需 |

**刻意延後**（與 `PRODUCT_SCOPE` Out of scope 一致）：登入、多租戶、題庫匯入、協作編輯——**唔影響上述假設驗證**，留待有明確付費方或合規要求再開。
